# v0.9.374 - Sprint 16: Repository Bridge, Runtime Health, and Durable SQLite Recovery

## Summary
- Introduced a canonical repository bridge so runtime subsystems can queue durable mutations when SQLite is busy or temporarily unavailable, then flush them when the repository recovers.
- Wired the live MCP status surface and unified server status to report repository health, journal depth, and recovery recommendations directly instead of relying on stale watcher logs.
- Hardened the analysis worker, session manager, runtime-table health sync, and graph-metric enrichment path so transient SQLite locks degrade to queued or skipped work instead of noisy failures or dropped bulk writes.

## Validation
- `node --check` passed on the repository bridge, worker analysis, runtime table health, session manager, and enrichment modules touched in this cycle.
- `get_server_status()` now reports `repository.health = healthy`, `queuedDurable = 0`, and `runtimeCodeFresh = true` after the restart.
- `get_recent_errors()` returned a clean runtime with no active watcher alerts after the restart and replay window.
