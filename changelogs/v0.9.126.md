## Service Boundary Policy Fixes

### Fix: Falsos Positivos en missing_compiler_service_bridge

**Problema detectado:**
- El compiler marcaba archivos de Layer C (Data Access) como violaciones de política
- Archivos en `/query/queries/`, `/query/apis/`, `/storage/repository/` DEBEN usar DB directamente
- Eran la API canónica, no código que debería usar APIs

**Solución:**
- `service-boundary-conformance.js`: Agregada función `isCanonicalQueryImplementation()`
- Excluye Layer C de la regla `missing_compiler_service_bridge`
- **Resultado:** 2 → 1 casos restantes (el restante es issue viejo que se limpia con FileWatcher)

### Nueva API: getFileExports

**Problema detectado:**
- `filesystem-validation.js` usaba `repo.db.prepare()` directamente
- Violaba política de service boundary
- No había API canónica para obtener exports de un archivo

**Solución:**
- `exports-query.js`: Nueva API con `getFileExports()` y `hasExport()`
- Usa `loadAtoms()` del storage layer en lugar de SQL directo
- Exportada através de: `atoms/index.js` → `file-query/index.js` → `file-api.js`

**Refactor:**
- `filesystem-validation.js`: Reemplazado `repo.db.prepare()` con `getFileExports()`
- Eliminadas 60+ líneas de SQL directo
- Ahora sigue patrón arquitectónico correcto

**Beneficio:**
- `filesystem-validation.js` ahora usa API canónica
- Fix automático del drift `missing_compiler_service_bridge`
- Sistema más consistente y mantenible

## FileWatcher Auto-Reconciliation

### Problema Crítico Detectado

**Síntoma:**
- Issues permanecían en la DB aunque los archivos se arreglaban
- FileWatcher no limpiaba issues resueltos automáticamente
- `context-resolver.js` aparecía como issue aunque ya usaba `findAtomByLine`

**Causa Raíz:**
- `reconcileWatcherIssues()` solo se ejecutaba al cargar issues (`loadWatcherIssues`)
- FileWatcher no disparaba reconciliación después de analizar archivos
- Issues viejos nunca se limpiaban sin intervención manual

**Flujo Roto:**
```
1. FileWatcher detecta cambio
2. Ejecuta handleFileModified()
3. Analiza archivo (analyzeFile)
4. Ejecuta guards (runImpactGuards)
5. ❌ NO limpia issues viejos ← BUG
```

**Flujo Corregido:**
```
1. FileWatcher detecta cambio
2. Ejecuta handleFileModified()
3. Analiza archivo (analyzeFile)
4. Ejecuta guards (runImpactGuards)
5. ✅ Limpia issues viejos (reconcileWatcherIssues) ← FIX
```

### Solución Implementada

**Archivo:** `src/core/file-watcher/handlers/file-handlers.js`

```javascript
// Reconcile watcher issues - clean up resolved issues automatically
try {
  const { reconcileWatcherIssues } = await import('../watcher-issue-persistence.js');
  const reconciliation = await reconcileWatcherIssues(this.rootPath, { maxDelete: 100 });
  if (reconciliation.deletedExpired || reconciliation.deletedSuperseded || reconciliation.deletedOutdated) {
    logger.info(
      `[ISSUE RECONCILE] Cleaned up ${reconciliation.deletedExpired + reconciliation.deletedSuperseded + reconciliation.deletedOutdated} resolved issue(s) after processing ${filePath}`
    );
  }
} catch (error) {
  logger.warn(`[ISSUE RECONCILE SKIP] Failed to reconcile issues after ${filePath}: ${error.message}`);
}
```

**Resultado Inmediato:**
- **5 issues viejos eliminados automáticamente**
- `deletedOutdated: 5` en `watcherReconciliation`
- Issue de `context-resolver.js` desapareció
- Sistema se mantiene limpio sin intervención manual
