# v0.9.186 - Sprint 16: Smart Batch Processor Complexity Split

## What changed
- Extracted the batch mode and batch execution control flow out of `SmartBatchProcessor` into module-local helpers.
- Renamed the generic `clear()` API to `resetBuffer()` to make the buffer lifecycle explicit and reduce conceptual duplication.
- Updated the file-watcher shutdown path to use the new buffer reset name.

## Why
- `SmartBatchProcessor` was still flagged as a high-complexity watcher hotspot.
- Moving the branching logic out of the class keeps the exported processor as a thinner orchestration surface.
- The `clear()` name was too generic and collided semantically with other lifecycle helpers in the project.

## Validation
- `node --check` passed for the updated batch processor and shutdown modules.
- `get_recent_errors()` showed only stale-runtime notifications for the updated files until the MCP runtime is reloaded.
