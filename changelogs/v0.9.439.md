# v0.9.439 - Pattern Detection Debt Cleanup & Snapshot Service Split

**Fecha:** 2026-04-11  
**Tipo:** Technical Debt Cleanup + Snapshot Governance  
**Impacto:** Medio-Alto

---

## Resumen

Esta versión concentra una tanda grande de limpieza estructural en la capa de pattern detection y en los servicios canónicos de snapshots/cache del compilador.

El objetivo fue bajar complejidad real, extraer lógica repetida a helpers puros, y dejar los orquestadores más delgados sin cambiar los contratos de salida. También se separó parte del armado de snapshots y del cache policy advisor para reducir deuda de mantenimiento en rutas críticas de observabilidad.

---

## Cambios Principales

### 1. Pattern Detection más modular

Se extrajo lógica repetida a helpers puros y se simplificaron detectores de la familia `layer-a-static/pattern-detection`:

- `deep-chains`
- `orchestration-gaps`
- `performance-patterns`
- `omnysys-health`
- `sql-patterns`
- `hotspots`
- `shared-objects` (`risk-analyzer`)

Los detectores quedaron como orquestadores finos y se añadieron tests unitarios específicos para mantener el contrato estable.

### 2. Snapshot service dividido por responsabilidades

Se partió `compiler-snapshot-service` en una capa de helpers para separar:

- ensamblado de contratos cross-surface
- composición final del snapshot canónico
- persistencia de contexto y observabilidad

Esto reduce la complejidad del entrypoint que usan `get_metrics_snapshot`, `get_health_snapshot`, `get_health_panel` y otros consumidores MCP.

### 3. Cache policy advisor refactorizado

Se extrajeron helpers de:

- resumen de recent errors
- resumen de tool telemetry
- hotspots recurrentes
- surfaces de cache/no-cache

El resumen de cache policy ahora mantiene el mismo contrato, pero con menos lógica monolítica en el archivo principal.

### 4. CLI MCP bridge corregido

Se corrigieron las rutas del CLI para que `status`, `tools` y `call` consulten el endpoint correcto del registry MCP.

Esto evitó lecturas engañosas del estilo `Tools: 0 available` cuando el runtime sí estaba vivo y el problema era solo de la capa de representación.

### 5. Coherencia de historia y trust

Se mantuvieron y alinearon las capas de:

- `health-history`
- `metrics-history`
- `compiler-persistence-paths`
- `system-inventory`
- `policy coverage`
- `trust investigation`

El sistema ahora separa mejor:

- verdad viva
- historia diaria
- síntesis canónica
- drift real
- drift de representación
