# v0.9.451 - MCP HTTP Summary, Cache-First Status, and Runtime Surface Consolidation

**Fecha:** 2026-04-17
**Tipo:** Runtime observability + cache-first surface consolidation
**Impacto:** Alto

---

## Resumen

Esta versión consolida la salida de arranque y las superficies vivas del MCP para que el sistema muestre un control plane más legible sin reconstruir de más.

El boot summary vuelve a imprimir una foto útil del runtime, `status` incorpora explícitamente `MCP HTTP`, y varias rutas críticas pasaron a reutilizar snapshots/caché viva antes de caer al camino algebraico pesado.

---

## Cambios Principales

### 1. Boot summary y control plane visibles

- El dashboard de arranque vuelve a imprimirse de forma estable durante `ready`.
- La salida inicial ahora muestra explícitamente la línea `MCP HTTP` con URL y puerto.
- `status` expone una fila propia para `MCP HTTP`, además de `Daemon`, `Proxy` y `Bridge`.
- El resumen de estado dejó de depender solo del envelope mínimo y ahora conserva las secciones canónicas relevantes del sistema.

### 2. Status e inventario cache-first

- `get_server_status()` y `get_system_inventory_report()` priorizan snapshots vivos/materializados antes de recomponer superficies pesadas.
- La ruta de inventario quedó más compacta por defecto y conserva el detalle solo cuando se pide.
- `loadCompilerExplainability()` y el snapshot service reutilizan la misma verdad viva para evitar duplicación de cómputo.

### 3. Propagación y runtime más consistentes

- La propagación ya no se trata como una explosión global ante cada cambio; ahora se apoya más en scope, caché y materialización incremental.
- El watcher, el orchestrator y el hot-reload comparten mejor el estado de cambio y de frescura.
- Se corrigieron rutas/imports rotos y contratos ESM que podían tumbar el worker al arrancar.

### 4. Salud operativa y telemetría

- `status`, `health` y el resumen de arranque muestran mejor el estado del daemon, del proxy y de las sesiones.
- Se mantuvo la telemetría de proxy/bridge para distinguir mejor HTTP direct de stdio bridge.
- El sistema ya no necesita reconstruir el dashboard completo para mostrar una foto útil al iniciar.

---

## Resultado Operativo

Estado visible tras la iteración:

- `status`: disponible y estable
- `ready`: imprime resumen completo al arrancar
- `MCP HTTP`: visible en boot summary y control plane
- `databaseHealth`: sigue siendo una surface viva, pero ya no bloquea el boot summary

---

## Notas

- El SQL/algebraic bypass sigue existiendo como fallback, pero la intención ahora es que no sea la ruta común.
- La deuda restante es principalmente de consolidación y unificación de superficies, no de capacidad funcional básica.
- Esta versión sigue empujando el sistema hacia `SQLite + materialized projections + cache-first reads + incremental propagation`.
