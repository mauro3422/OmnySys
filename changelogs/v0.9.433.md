# v0.9.433 - Sprint 16c: MCP Bridge/Proxy Restart Stabilization (2026-04-09)

## Key Achievements

1. Hardened the MCP stdio bridge restart/reconnect path so restart ACKs, session promotion, and recovered transports no longer depend on the old pre-restart socket state.
2. Fixed the proxy bootstrap/watchdog path so a proxy that discovers an already-live daemon no longer falls through and spawns a competing worker.
3. Added `alive` daemon semantics for startup/ownership decisions, preventing secondary proxies from treating a booting daemon as dead just because `/health` is still in `starting`.
4. Added finer worker crash tracing for `orchestrator-init` and `ready-step`, so future `code=1` exits record the exact initialization subphase instead of failing as blind crashes.

## Impact

* MCP reconnects are less likely to collapse into `Transport closed` after restart or cold bridge bootstrap.
* Duplicate proxy ownership is less likely to create hidden worker contention on port `9999`.
* Future worker exits during bootstrap now leave enough context to distinguish `orchestrator-init`, `diagnostics-dashboard`, and final `mark-initialized` failures without guessing from timing alone.

## Known Follow-up

* A separate worker lifecycle bug is still open: some workers exit with `code=1` roughly 30-40 seconds after startup, even after reaching `ready`. See `docs/bugs/mcp-worker-periodic-exit-after-ready.md`.
