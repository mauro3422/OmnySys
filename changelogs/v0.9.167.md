# v0.9.167 - Sprint 16: System Map Coverage and Dependency Repair Heuristics

**Date:** 2026-03-21

## Summary

This release tightens the system-map repair path so coverage checks compare `system_files` against the active file universe instead of historical atom file paths. The dependency repair logic was also split into a dedicated helper, keeping the main repair orchestrator under the watcher threshold while preserving DB-only recovery for `file_dependencies`.

## Key Changes

1. **Coverage Heuristic Fix**: `getSystemMapPersistenceCoverage` now measures `system_files` lag against active files, eliminating a false drift warning that came from comparing against historical atom file paths.
2. **Dependency Repair Split**: `repairFromSystemFileDependsOn` now lives in a dedicated helper so `system-map-persistence-repair.js` stays compact and easier to audit.
3. **DB-Only Validation**: the canonical health path still resolves support-table state from SQLite only; no filesystem fallback was added.

## Validation

- `node --check` passes for the updated system-map persistence modules.
- Direct SQLite verification shows `system_files` and `files` are aligned on active rows and `file_dependencies` is populated again.
- `check_pipeline_integrity` returns `A+` after the repair path runs.
