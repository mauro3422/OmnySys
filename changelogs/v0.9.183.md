# v0.9.183 - Sprint 16: Canonical Metadata Surface Gateway

## What changed
- Added a canonical metadata gateway for file, batch, and atom surfaces so callers no longer need to know which extraction entrypoint to invoke.
- Redirected the main file-level and batch-level pipeline entrypoints to the gateway.
- Kept atom extraction on the rich atom pipeline, but wrapped it in the same surface contract so the output carries an explicit surface kind.
- Simplified the metadata gateway and the enhance orchestrator to reduce structural complexity.

## Why
- OmnySys had multiple metadata entrypoints that were correct individually but too easy to mix up from the outside.
- The gateway gives a single, explicit contract for file, batch, and atom metadata surfaces while keeping the existing extractors intact.
- This reduces future ambiguity for AI agents and keeps the DB-first contract visible at the pipeline boundary.

## Validation
- `node --check` passed for the touched pipeline and runtime files.
- `vitest` passed for the metadata gateway regression and the existing pipeline-group regression.
- `get_recent_errors()` no longer reports active high-complexity debt for the `enhance` orchestrator after the split, only the expected remaining medium/low alerts.
