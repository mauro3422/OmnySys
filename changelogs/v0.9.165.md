# v0.9.165 - Sprint 16: Dead-Code Core Split and Taxonomy Extraction

**Date:** 2026-03-21

## Summary

This release splits the dead-code helper surface into smaller canonical modules for taxonomy, normalization, suspicion, SQL predicates, drift detection, and remediation reporting. The public API stays stable through the existing barrel surfaces, but the internal responsibilities are now separated so the watcher and compiler policies have a smaller, clearer target.

## Key Changes

1. **Taxonomy Extraction**: dead-code naming, purpose, file, and canonical-signal patterns now live in a dedicated taxonomy module.
2. **Normalization Split**: atom normalization stays separate from suspicion logic, reducing the size of the main helper surface.
3. **Suspicion Module**: the suspicious-dead-code decision is isolated into its own file so the heuristics remain easier to audit.
4. **Reporting Split**: SQL counting, plausibility summaries, candidate loading, and remediation plan generation moved into a dedicated reporting module.
5. **Barrel Compatibility**: `dead-code-utils.js` remains the compatibility entrypoint, so existing imports keep working without API churn.
6. **Regression Coverage**: the dead-code utility regression test now covers normalization, suspicion, plausibility, and remediation plan generation.

## Validation

- `node --check` passes for the new dead-code modules and the compatibility barrel.
- The new Vitest regression file passes.
- The runtime only reports stale-module reload for the updated dead-code barrel until the next MCP restart; no new active watcher alerts were introduced.
