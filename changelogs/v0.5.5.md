# v0.5.5 - Error Handling & Async Safety Improvements

## Fecha: 2026-04-02

## Problema
El sistema tenía un repair rate bajo (27%) debido a:
1. `detect-folderization-opportunities.js`: 6 átomos sin error handling (CRITICAL risk)
2. `split-large-file.js`: Warning de async safety (network calls sin error handling)
3. Thrashing runs: 60 intentos de reparación fallidos

## Solución

### 1. Error Handling en `detect-folderization-opportunities.js`
Agregado try/catch granular en `detectHighCoupling()`:
- Query de files: try/catch con logger.error
- Query de atoms: try/catch con logger.warn + continue
- Query de call_graph: try/catch con logger.warn + continue
- Análisis de coupling: try/catch con logger.warn + continue

Resultado: 6 átomos sin error handling → 0 átomos sin error handling

### 2. Async Safety en `split-large-file.js`
Agregado try/catch granular en `analyzeFileForSplit()`:
- Query de atoms: try/catch con logger.error + throw
- Query de call_graph: try/catch con logger.error + throw
- Lectura de archivo: try/catch con logger.error + throw
- Extracción de imports: try/catch con logger.warn + fallback []
- Agrupación de átomos: try/catch con logger.error + throw

Resultado: Warning de async safety eliminado

### 3. Sistema de Repair Rate

#### Métricas antes:
- repairRateOnPressure: 27% (7/26 pressured runs)
- repairYield: 10% (7/70 total repair attempts)
- thrashingRuns: 60 (intentos fallidos)
- repairScore: -1.47 (score negativo)

#### Métricas esperadas después:
- repairRateOnPressure: 42%+ (mejora por error handling)
- repairYield: 15%+ (menos thrashing)
- thrashingRuns: 40- (menos intentos fallidos)
- repairScore: -0.5+ (score menos negativo)

#### Cómo funciona el Repair Rate:
```
TOOL RUN → Detecta problema (pressure) → Intenta reparar → SUCCESS/FAIL
                                                          ↓
                                                   THRASHING (si falla)
```

- **pressureRuns**: Ejecuciones que detectaron problemas
- **repairedRuns**: Reparaciones exitosas
- **thrashingRuns**: Intentos de reparación fallidos
- **repairRateOnPressure**: repairedRuns / pressureRuns
- **repairYield**: repairedRuns / (repairedRuns + thrashingRuns)

### 4. Metadatos del Schema Disponibles para Monitoreo

El schema de átomos tiene 43 campos útiles:
- `fragilityScore`: Fragilidad (0-1) - predictor de fallos
- `changeRisk`: Riesgo de cambio (0-1) - priorización
- `centralityScore`: Centralidad en grafo - nodos críticos
- `changeFrequency`: Frecuencia de cambios - código inestable
- `complexity`: Complejidad ciclomática
- `couplingScore`: Acoplamiento

Correlaciones útiles:
- `calls ↔ externalCallCount`: r=0.99 (predictivo)
- `complexity ↔ linesOfCode`: r=0.64 (detectar god functions)
