# v0.9.101 - Scanned File Manifest Unification

Date: 2026-03-06

## Summary

The compiler now persists a canonical scanned-file manifest so scanner discovery, hash-cache change detection and persisted coverage telemetry all work against the same file universe, including files that intentionally produce `0` atoms.

## Highlights

- Added canonical manifest helpers in `src/shared/compiler/compiler-persistence.js`:
  - `getPersistedScannedFilePaths`
  - `getPersistedKnownFilePaths`
  - `syncPersistedScannedFileManifest`
  - `summarizePersistedScannedFileCoverage`
- Updated `file-scanner.js` and `change-detector.js` to use the manifest before comparing scanner/hash/index state.
- Extended `status.js` and `pipeline-health-handler.js` so explainability now exposes `persistedFileCoverage`.
- Added standardization/guidance support for the `scanned_file_manifest` pattern.

## Result

- Zero-atom files remain first-class members of the compiler file universe.
- The scanned-file manifest synchronized to `2165/2165` files during verification.
- The previous “Recovered X file(s) present in hash cache but missing from the persisted index” drift is resolved through canonical sync rather than repeated recovery noise.
