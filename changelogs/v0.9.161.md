# v0.9.161 - Sprint 16: Framework Hook Duplicate Policy

**Date:** 2026-03-21

## Summary

This release teaches the duplicate-signal policy to treat canonical MCP `performAction` hooks as intentional framework contracts instead of conceptual duplicate noise. The policy now has an explicit framework-coordinator check so the watcher can distinguish shared framework hooks from accidental repeated business logic.

## Key Changes

1. **Framework Hook Policy**: `duplicate-signal-policy` now recognizes canonical MCP tool/base-tool `performAction` hooks as an intentional coordinator surface.
2. **Heuristic Boundary**: the new policy is narrow and only exempts the canonical MCP framework paths, not arbitrary `performAction` functions elsewhere.
3. **Regression Coverage**: a unit test now verifies that MCP tool hooks are ignored while non-framework `performAction` surfaces remain actionable.

## Validation

- The policy helper passes `node --check`.
- The new unit regression covers both the framework and non-framework cases.
- This change only adjusts duplicate heuristics; it does not affect DB canonicality or tool execution behavior.
