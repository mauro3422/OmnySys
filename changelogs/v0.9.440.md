# v0.9.440 - MCP Restart/Recovery Stability and Trust Gate Cleanup

**Fecha:** 2026-04-11  
**Tipo:** MCP Stability + Trust Governance  
**Impacto:** Medio

---

## Resumen

Esta versión consolida la estabilización del bridge MCP y la limpieza de la capa de confianza del compilador.

El foco fue separar ruido de observabilidad de señales reales de gobernanza, corregir rutas rotas en utilidades de `scripts/`, y dejar documentada la frontera entre reinicio, compaction y pérdida temporal de sesión en el daemon MCP.

---

## Cambios Principales

### 1. Utilidades de scripts reencaminadas

Se corrigieron rutas y referencias rotas en utilidades operativas para que los checks locales vuelvan a apuntar a los archivos reales dentro de `scripts/utils/` y `scripts/diagnostics/`.

Esto evitó falsos fallos en `npm run check` y dejó claro que estas utilidades son wrappers de mantenimiento, no parte del runtime canónico.

### 2. Trust gate menos ruidoso

Se ajustaron las señales de `trust` para que:

- `metadata_coverage` use la cobertura de campo efectiva
- `issue_persistence` no castigue ruido histórico aislado
- `tool_latency` quede como señal observacional
- `policy_drift` siga siendo visible sin bloquear por umbrales artificiales

El resultado es una superficie de confianza más honesta y menos sensible a falsos bloqueos.

### 3. Live-row sync y drift de fase 2

Se suavizó el impacto de `phase2_settling` en la sincronización de live rows y en el assessment de drift, para no marcar como bloqueante una ventana transitoria de estabilización.

### 4. Documentación de bugs MCP

Se revisó y alineó la evidencia local de:

- restart/reconnect
- startup false negatives
- WSL launcher freeze
- periodic worker exits after ready

La conclusión práctica es que un `transport closed` o un schema/tool registry ausente durante compaction o restart no implica necesariamente una rotura permanente del bridge.

## Recomendación

Si el problema reaparece, responder en este orden:

1. verificar si hubo compaction o restart MCP en esa ventana
2. pedir `get_server_status` y `get_recent_errors`
3. distinguir entre daemon sano, bridge sano y sesión realmente adoptada
4. no asumir que una ausencia temporal de tools es una pérdida permanente del schema
