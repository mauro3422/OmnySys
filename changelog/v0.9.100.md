# v0.9.100 - Session Recovery, Drift Reconciliation & SQLite Snapshot Alignment

**Fecha:** 2026-03-28  
**Tipo:** Runtime Stability + Canonical Drift + Legacy Snapshot Alignment  
**Impacto:** Alto

---

## Resumen

Esta versión cierra varias fuentes de drift que estaban repartidas entre el bridge MCP, la reconciliación de watcher alerts y las superficies de snapshot del compilador.

El foco fue estabilizar la identidad de sesión y la recuperación del bridge, formalizar una evaluación canónica de drift para el snapshot/status, y alinear la herramienta offline legacy con la fuente de verdad actual en SQLite.

---

## Cambios Principales

### 1. Recovery y Affinity de Sesiones MCP

- Se endureció la recuperación del bridge stdio para evitar churn entre `reconnecting` y sesiones recreadas innecesariamente.
- El routing HTTP ahora tolera una ventana corta de gracia antes de declarar `SESSION_EXPIRED`.
- La identidad canónica de cliente quedó alineada sobre `client_id` para evitar buckets duplicados por `name`.

### 2. Reconciliación de Watcher Alerts

- Las alertas de `code_file_size_*` ahora se reconcilian contra el estado real del archivo.
- Cuando el archivo vuelve a estar por debajo del umbral guardado en el contexto del issue, la alerta queda obsoleta y deja de contaminar `_recentErrors`.
- Esto evita que el watcher quede pegado a una foto vieja después de que la DB y el grafo ya se actualizaron.

### 3. Snapshot Canónico de Drift

- Se formalizó una evaluación canónica de drift para el compilador y sus superficies de salud.
- El snapshot/status ahora expone señales unificadas para:
  - `policy_drift`
  - `analysis_generation`
  - `data_gateway_contract`
  - `live_row_sync`
  - `system_map_persistence`
  - `semantic_surface_granularity`
  - `file_universe_granularity`
- La health view deja de depender de heurísticas dispersas para interpretar esas señales.

### 4. Legacy Offline Tool Alignment

- `scripts/omny-tools.js` quedó corregido para importar el barrel real de comandos.
- El loader offline dejó de exigir `atoms.json` y ahora lee primero desde la SQLite canónica en `.omnysysdata/omnysys.db`.
- `server-status` ahora degrada de forma limpia cuando no hay datos y no rompe por divisiones entre cero.

---

## Resultado Operativo

Validaciones locales realizadas:

- El CLI offline `node scripts/omny-tools.js status` vuelve a arrancar correctamente.
- El status offline ya reporta datos reales desde SQLite:
  - `Átomos`: `8957`
  - `Archivos en system-map`: `2495`
- Los cambios de sesión/recovery y de drift quedaron cubiertos por tests unitarios ya ejecutados en la iteración.

---

## Estado

Lo que queda mejor:

- Menos churn en reconexión MCP.
- Menos falsos positivos de watcher por snapshots viejos.
- Un status/diagnóstico más confiable para la capa de compilador y data gateway.
- La herramienta offline ya no depende de archivos JSON legacy que no forman parte del flujo actual.

Lo que sigue abierto:

- Seguir consolidando superficies legacy que todavía vivan fuera de la fuente canónica.
- Reducir deuda estructural en módulos grandes del runtime que siguen por encima del umbral ideal.
- Seguir empujando consumidores residuales hacia helpers/snapshots canónicos del sistema.
