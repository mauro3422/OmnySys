# v0.9.83 — Daemon Stability & Semantic Pipeline Fix

**Date:** 2026-03-04  
**Type:** Bug Fixes + Stability  
**Status:** ✅ Released

---

## Summary

This version fixes a cascade of 7 bugs that were preventing: the MCP daemon from surviving restarts, semantic connections from ever being persisted to SQLite, and multiple MCP tools from returning correct data.

---

## Bugs Fixed

### 🛡️ Bug 1 — Daemon kills itself on `restart_server`
**Root cause:** The component restart path always ran `LayerAAnalysisStep` (~10-30s). During that time `server.initialized = false`, causing VS Code to detect the task as "completed" and re-launch it. The new process found port 9999 busy and exited, breaking the MCP connection permanently.

**Fix:** `restart-server.js` — default restart now only restarts the Orchestrator (<1s). Layer A only runs when `reanalyze: true` is explicitly passed. The MCP HTTP server stays alive throughout.

**New restart modes:**
| Option | Time | Effect |
|--------|------|--------|
| `clearCacheOnly: true` | <100ms | Flush RAM cache + tool registry |
| `clearCache: true` (default) | <1s | Cache + Orchestrator restart |
| `reindexOnly: true` | ~10s | Force Layer A (DB preserved) |
| `reanalyze: true` | ~30s | Wipe DB + full Layer A reindex |

---

### 🔑 Bug 2 — `aggregate_metrics: risk` returned random keys
**Root cause:** `risk-handler.js` used `_category_${category}_${Math.random()}` as fallback `file_path` when a risk item had no real path. This created garbage keys like `_category_scores_0.502...` in `risk_assessments`.

**Fix:** Skip items without a real `file_path` entirely — they're metadata blobs, not file risks.

---

### 📊 Bug 3 — `status.metadata.totalFunctions` desync
**Root cause:** `status.js` read from in-memory metadata cache which could be stale.

**Fix:** Added `liveAtomCount` and `liveFileCount` via direct `SELECT COUNT(*)` from SQLite atoms table.

---

### 🔬 Bug 4 — `health` returned 0/0 for all physics vectors
**Root cause:** `aggregate-metrics.js` health case returned hardcoded zeros.

**Fix:** Direct SQLite query: `SELECT COUNT(*), AVG(fragility_score), AVG(coupling_score), AVG(cohesion_score), AVG(importance_score), AVG(centrality_score) FROM atoms`. Now returns real values like `avgCoupling: 0.131, avgCohesion: 0.397`.

---

### 🔌 Bug 5 — HotReloadManager crashes on every file save
**Root cause:** `ReloadHandler._executeStrategy(filename, moduleInfo)` called `moduleInfo.type` without null guard. `ModuleClassifier.classify()` returns `null` when a file doesn't match any reloadable pattern — crashing with `Cannot read properties of undefined (reading 'name')`.

**Fix:** `reload-handler.js` — null guard: if `moduleInfo === null`, skip silently. Also: `HotReloadManager.index.js` already had this guard — the crash was in a secondary code path.

---

### 💾 Bug 6 — `aggregate_metrics: patterns` always returned 0 semantic connections
**Root cause (chain of 5 issues):**

1. `system-map-enhancer.js` stored 264 connections in `enhanced.connections.{sharedState, eventListeners, routes...}` (structured object) but `persistSystemMapToDb` read `systemMap.semanticConnections` (flat array). Key mismatch = nothing ever reached SQLite.

2. `saveSemanticData` was declared `async` but called synchronously inside a `db.transaction()` callback (better-sqlite3 transactions are fully synchronous). The function body was scheduled as a microtask AFTER the transaction committed — the inserts ran outside the transaction.

3. `semantic_issues` INSERT had `NOT NULL constraint failed: semantic_issues.file_path` — some issues had `undefined` filePath. This caused the **entire transaction to rollback**, silently wiping `semantic_connections` too.

4. `SQLiteAdapterCore.saveSystemMap()` was synchronous but called `async persistSystemMapToDb` without awaiting it — unhandled rejection when the transaction threw.

5. `aggregate-metrics.js` patterns case queried with `this.repo?.db` — but `this.repo` was never `null` (GraphQueryTool.execute initializes it), however the block logic was using `getRepository(this.projectPath).db` which was the correct fix.

**Fixes applied:**
- `system-map-enhancer.js` — adds `enhanced.semanticConnections` flat array alongside the structured `.connections` object
- `system-map.js` — `persistSystemMapToDb` normalizes both shapes (flat array OR structured `.connections.*` object)
- `semantic-handler.js` — removed `async` from `saveSemanticData`, `loadSemanticConnections`, `loadSemanticIssues` (all calls are sync better-sqlite3)
- `semantic-handler.js` — filters `semantic_issues` to only entries with a valid `filePath` before INSERT
- `sqlite-adapter-core.js` — `saveSystemMap` is now `async` and properly `await`s `persistSystemMapToDb`
- `save.js` — `saveEnhancedSystemMap` now `await`s `repo.saveSystemMap()`
- `aggregate-metrics.js` — patterns case uses `this.repo?.db` (set by GraphQueryTool.execute)

**Result verified live:** `semanticConnections.total: 264` with breakdown by type.

---

### 🛠️ Bug 7 — `restart_server` schema missing new options
**Fix:** `tools/index.js` — added `clearCacheOnly` and `reindexOnly` to the MCP tool inputSchema so agents can discover and use them correctly.

---

### 🏙️ Bug 8 — `aggregate_metrics: societies` returned 0 clusters
**Root cause (chain of 3 issues):**
1. `SocietyEngine.buildSocieties()` queried `repo.getAll({ limit: 0 })` which translates exactly to `LIMIT 0` in SQLite.
2. `adapters/helpers/converters.js` function `rowToAtom(row)` was missing a `return enriched;` statement, mapping the entire 11k row result into an array of `undefined`.
3. Race condition: `static-insights.js` executing `analyzeSocieties()` immediately at boot before the deep background indexer repopulates the SQLite Database during a hard wipe (`reanalyze=true`), persisting an empty cluster snapshot indefinitely.

**Fix:** Removed the `limit: 0` constraint, added the `return` statement in the SQLite row converter, and instructed the user that soft-restarts (`reanalyze: false`) are required to populate the society table natively once the DB is filled.

---

### 🧮 Bug 9 — `avgFragility` and `avgCentrality` physics returned 0 globally
**Root cause:** `atom-enricher.js` tried to map `vectors.fragility` instead of the correctly computed `vectors.fragilityScore`. Meanwhile, `avgCentrality` didn't have an algorithm to compute during the single-file Phase 1 extraction because the global `context.callers` tree map was empty.

**Fix:** Fixed key map assignments in the enricher. Implemented `calculateCentrality` in `semantic-calculator.js` with intelligent fallbacks pointing to the local `atom.calledBy` and `atom.calls` graphs.

---

### 💥 Bug 10 — Background Hot-Reload Worker crashed reading `.name`
**Root cause:** The file-watcher incremental extractor `handleRemovedAtoms(atoms)` parses the existing SQLite database atoms to preserve existing data. Because `rowToAtom` (from Bug 8) was piping `undefined` records into SQLite's query response, the filter `.filter(a => a.name)` triggered fatal worker exceptions.

**Fix:** Completely solved automatically by restoring the `return enriched;` statement in `converters.js` from Bug 8.
